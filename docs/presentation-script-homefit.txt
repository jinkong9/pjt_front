HOME FIT 발표 스크립트 보강안

전제 멘트
현재 서비스는 전체 기능이 완전히 고도화된 최종 버전이라기보다는, 공공데이터와 사용자 데이터를 하나의 주거 의사결정 흐름으로 연결하는 MVP 단계입니다. 그래서 발표에서는 "지금 구현된 핵심 흐름"과 "확장 가능한 구조"를 중심으로 설명하겠습니다.


[11번 슬라이드] 실거래 매물

이 화면은 HOME FIT의 가장 기본이 되는 실거래 매물 탐색 기능입니다.

SSAFY에서 제공받은 아파트 실거래 데이터를 기반으로 하고, 여기에 공공데이터 포털에서 가져온 오피스텔과 원룸 실거래 데이터까지 함께 연결해서 사용자가 다양한 주거 유형을 한 번에 확인할 수 있도록 구성했습니다.

기존에는 아파트는 아파트대로, 원룸이나 오피스텔은 다른 플랫폼에서 따로 찾아봐야 했는데, HOME FIT에서는 지도 위에서 지역을 기준으로 거래 정보를 확인하고, 주거 유형별로 비교할 수 있습니다.

기술적으로는 프론트에서 TanStack Query를 사용해서 시도, 구군, 동, 그리고 유형별 실거래 요청을 캐싱합니다. 같은 지역을 반복 조회할 때 매번 서버에 새 요청을 보내지 않고, 필요한 시점에만 데이터를 갱신하도록 설계했습니다.

백엔드에서는 실거래 데이터를 한 번 조회할 때마다 외부 API에 의존하기보다, 배치 구조를 통해 공공데이터를 정제해서 저장하고, 사용자는 저장된 데이터를 빠르게 조회하는 방향으로 구성했습니다.


[12번 슬라이드] 대출 상품 & 월 납부 계산

이 기능은 단순히 "이 집이 얼마인지"를 보는 데서 끝나는 것이 아니라, 사용자가 실제로 감당할 수 있는 집인지 판단할 수 있도록 만든 기능입니다.

금융감독원 오픈 API를 통해 주택담보대출과 전세자금대출 관련 상품 정보를 가져오고, 사용자가 입력한 보유 자산, 연 소득, 월 가용 금액을 기준으로 예상 월 이자 부담을 계산합니다.

예를 들어 매물 가격에서 보유 자금을 제외한 필요 대출 금액을 계산하고, 상품별 금리 조건을 적용해서 월 납부액이 사용자의 월 가용 금액 안에 들어오는지 판단할 수 있게 했습니다.

현재는 완성도를 계속 다듬는 단계지만, 핵심 의도는 실거래 탐색과 금융 계산을 분리하지 않고 하나의 흐름으로 연결하는 것입니다.

기술적으로는 금융감독원 API에서 가져온 대출 상품을 백엔드 배치 서비스로 저장하고, 프론트에서는 TanStack Query와 mutation을 이용해 금융 프로필 저장, 대출 분석 요청, 결과 갱신을 관리합니다.


[13번 슬라이드] LH 공고

이 화면은 공공데이터에서 가져온 공공임대와 LH 공고를 한눈에 볼 수 있도록 만든 기능입니다.

사용자는 지역, 유형, 모집 상태를 기준으로 공고를 확인할 수 있고, 로그인한 뒤에는 관심 공고로 저장할 수 있습니다. 이후 마이데이터에 입력한 소득, 자산, 월 저축액 같은 조건을 활용해서 사용자에게 더 적합한 공고를 추천하는 흐름으로 이어집니다.

또한 조건에 맞는 공고나 관심 등록한 공고가 있을 경우 이메일을 발송할 수 있도록 구성했습니다. 특히 신청 기간이 임박한 공고를 사용자가 직접 기억하지 않아도 알림을 받을 수 있게 하는 것이 목적입니다.

기술적으로는 LH 공고 데이터를 스케줄러와 배치를 통해 주기적으로 수집하고, DB에 캐싱해 둔 데이터를 사용자에게 제공합니다. 이메일은 Spring Scheduler가 정해진 시간에 관심 공고와 추천 공고를 검사하고, JavaMailSender를 통해 발송하는 구조입니다.

프론트에서는 TanStack Query로 LH 공고 목록, 상세, 관심 등록, 추천 공고 데이터를 관리하고, 관심 등록 이후에는 관련 쿼리를 invalidate해서 화면 상태가 바로 갱신되도록 했습니다.


[15번 슬라이드] LH 캘린더

LH 공고는 단순히 목록만 확인하는 것으로는 부족하다고 생각했습니다. 공고마다 접수 시작일, 진행 중인 기간, 마감일이 다르고, 사용자가 여러 공고를 동시에 보면 일정을 놓치기 쉽기 때문입니다.

그래서 앞서 설명한 LH 공고들을 캘린더 형태로 한눈에 볼 수 있게 만들었습니다. 사용자는 월간 캘린더에서 접수 시작, 공고 진행 중, 마감 일정을 날짜 기준으로 확인할 수 있습니다.

이 기능은 특히 공공임대를 준비하는 사용자에게 중요합니다. 좋은 공고를 찾는 것만큼, 신청 기간을 놓치지 않는 것도 실제 주거 선택에서 큰 영향을 주기 때문입니다.

기술적으로는 백엔드에서 배치로 수집한 LH 공고의 일정 데이터를 저장하고, 프론트에서는 TanStack Query로 공고 목록을 받아 캘린더 이벤트로 가공합니다. 또한 스케줄러 기반 이메일 알림과 연결해서 마감이 가까운 공고를 사용자에게 알려주는 흐름까지 확장했습니다.


[16번 슬라이드] 양도 게시판

양도 게시판은 빠르게 집을 구해야 하는 사용자를 위해 고안한 기능입니다.

대학생 추가 합격, 인턴 합격, 갑작스러운 근무지 이동처럼 짧은 시간 안에 거주지를 찾아야 하는 경우가 많습니다. 하지만 일반적인 부동산 계약은 기본 기간이 6개월 이상이거나 초기 비용이 커서, 단기간 거주가 필요한 사람에게는 부담이 됩니다.

그래서 HOME FIT에서는 보증금, 월세, 관리비, 양도비, 입주 가능일, 남은 계약 기간을 한눈에 비교할 수 있는 양도 게시판을 제공했습니다. 사용자는 빠른 입주가 가능한 매물을 조건 중심으로 탐색할 수 있습니다.

이 기능에서 S3는 게시글 이미지 저장에 사용됩니다. 사용자가 양도 매물을 등록할 때 이미지를 서버에 직접 쌓아두는 것이 아니라, S3에 저장하고 DB에는 이미지 URL을 연결하는 방식으로 확장성을 고려했습니다.

프론트에서는 양도 게시글 목록, 상세, 작성, 수정, 삭제, 관심 등록을 TanStack Query의 useQuery와 useMutation으로 관리하고, 등록이나 수정 이후 관련 목록을 갱신하도록 구성했습니다.


[17번 슬라이드] 생활권 분석

생활권 분석은 사용자가 단순히 집 안의 조건만 보는 것이 아니라, 실제로 그 동네에서 생활할 때 편한지 판단할 수 있도록 만든 기능입니다.

카카오 지도 API를 통해 주변 편의시설을 조회하고, 병원, 마트, 약국, 카페, 학교 같은 생활 인프라를 반경별로 보여줍니다. 사용자는 500m, 1km, 1.5km처럼 범위를 선택해서 주변 환경을 확인할 수 있습니다.

또한 Open API와 지오코딩을 활용해서 버스 정류장과 지하철역 정보도 함께 확인할 수 있도록 했습니다. 단순히 "근처에 뭐가 있다"가 아니라, 교통 접근성과 생활 편의성을 함께 판단하는 구조입니다.

백엔드에서는 카카오 로컬 API와 교통 관련 공공데이터를 연동하고, 버스 정류장 정보는 스케줄러와 배치 구조로 미리 수집해 조회 성능을 높이는 방향으로 설계했습니다.

프론트에서는 생활권 분석 요청을 TanStack Query로 관리해서, 사용자가 반경이나 주소를 바꿨을 때 필요한 데이터만 다시 불러오도록 했습니다.


[20~21번 슬라이드에서 기술적으로 덧붙일 멘트]

마지막으로 기술 구조를 간단히 정리하면, HOME FIT은 Vue 3 프론트와 Spring Boot 백엔드로 구성되어 있습니다.

사용자가 화면에서 데이터를 요청하면 Vue 3에서 TanStack Query가 먼저 캐시를 확인하고, 필요한 경우 Axios를 통해 Spring Boot API를 호출합니다. 백엔드에서는 Spring Security와 JWT로 사용자 인증을 처리하고, MyBatis를 통해 MySQL 데이터를 조회합니다.

공공데이터는 사용자가 요청할 때마다 외부 API를 직접 호출하는 방식이 아니라, Scheduler와 Batch를 통해 LH 공고, 실거래, 대출 상품, 버스 정류장 정보를 주기적으로 수집하고 정제한 뒤 DB에 저장하는 구조입니다.

이미지 파일은 양도 게시판을 중심으로 S3에 저장하도록 구성했고, 메일 기능은 관심 공고와 추천 공고를 기준으로 스케줄러가 자동 발송할 수 있게 설계했습니다.

즉, HOME FIT은 단순 조회 서비스가 아니라, 공공데이터 수집, 사용자 조건 저장, 관심 등록, 추천, 알림까지 이어지는 통합 주거 의사결정 플랫폼을 목표로 하고 있습니다.
